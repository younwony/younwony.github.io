## 포맷 1
```
세 줄 요약

프로젝트는 “참여 기록”이 아니라 문제 해결 이력서로 작성해야 합니다.

핵심은 문제 → 판단 → 실행 → 성과의 일관된 구조입니다.

상위 20% 프로젝트만 남기고, 나머지는 과감히 정리하는 것이 경쟁력을 만듭니다.

상세 설명
1️⃣ 기본 원칙: 프로젝트는 “설명서”가 아니라 “성과 보고서”

7년차 이상에서 가장 흔한 실수는 무엇을 만들었는지를 길게 쓰는 것입니다.
면접관과 리뷰어는 이미 기능 구현 자체에는 관심이 없습니다.
다음 세 가지가 반드시 드러나야 합니다.

왜 이 문제가 중요했는가 (비즈니스/운영 관점)

그중 본인의 판단과 선택은 무엇이었는가

결과적으로 무엇이 얼마나 개선되었는가 (정량 지표)

2️⃣ 추천 프로젝트 단일 포맷 (가장 범용적)

아래 포맷은 커머스·마케팅·플랫폼 개발 전반에 재사용 가능한 구조입니다.

✅ 프로젝트 제목

네이버 쇼핑 API 연동 기반 가격 자동 조정 시스템 구축

✅ 프로젝트 한 줄 요약 (필수)

외부 가격 경쟁 데이터 수집·분석을 자동화하여 가격 경쟁력 확보 및 매출 10% 상승

✅ 배경 / 문제 정의 (Why)

수작업 가격 모니터링으로 MD/운영 리소스 과다 소모

네이버 쇼핑 API 호출 제한으로 실시간 대응 불가

가격 조정 기준이 불명확하여 운영 의사결정 일관성 부족

👉 이 단계에서는 “불편했다”가 아니라 조직의 손실을 명확히 씁니다.

✅ 해결 전략 및 설계 판단 (How – 핵심)

최저가 상품만 선별하여 API 호출 대상 최소화 (호출량 80% 절감)

모델 번호 기준 외부 가격 수집 → 내부 가격 정책 룰 엔진 적용

±2% 이내 경쟁 시 자동 가격 조정, 초과 시 알림 기반 수동 승인

배치 + 비동기 처리 구조로 API Rate Limit 안정화

👉 여기서 가장 중요합니다.
**“왜 이 방식이 합리적인 선택이었는지”**가 드러나야 합니다.

✅ 본인 역할 (What I did)

가격 조정 정책 로직 설계 및 기준 수립 주도

외부 API 제약 분석 및 호출 전략 설계

대용량 가격 데이터 정합성 검증 로직 구현

MD/마케팅팀과 정책 협의 및 운영 기준 문서화

👉 단순 구현 나열 ❌
판단·조율·주도를 강조합니다.

✅ 성과 및 임팩트 (Result)

가격 조정 자동화로 운영 업무 40% 감소

가격 경쟁력 확보로 대상 상품 매출 10% 증가

가격 이슈 대응 시간 평균 2시간 → 10분 이내 단축

👉 수치는 적어도 1개 이상 필수입니다.

3️⃣ 파레토 관점 정리 (매우 중요)
🔥 반드시 남겨야 할 상위 20%

매출, 비용, 속도, 안정성 등 비즈니스 지표가 변한 프로젝트

설계 선택이 드러나는 프로젝트

타 부서와 협업 구조를 바꾼 사례

✂️ 과감히 줄여야 할 하위 20%

CRUD 중심 내부 관리 도구

성과 없는 단순 유지보수

“~을 사용했다” 위주의 기술 나열 프로젝트

👉 프로젝트 수가 많을수록 오히려 약해집니다.
5~6개가 가장 이상적입니다.

4️⃣ 추천 루틴 (지금 당장 도입)

프로젝트마다 “이게 없었다면 조직이 잃는 것”을 한 줄로 정리

수치가 없으면 로그·리포트·GA·BigQuery부터 뒤져서 만든다

기술 스택은 본문이 아니라 결과를 설명하기 위한 수단으로만 사용
```

## 포맷 3

```
핵심 구조: STAR 기반 + 정량적 성과 + 기술 스택을 조합한 포맷이 가장 효과적입니다.

권장 프로젝트 기술 포맷
markdown### 프로젝트명 (기간)
> 한 줄 요약: 프로젝트의 핵심 가치를 한 문장으로

**상황(Context)**
- 비즈니스 배경 및 해결해야 할 문제
- 프로젝트 규모 (팀 구성, 데이터 규모 등)

**역할(Role)**
- 담당 범위 및 책임

**핵심 구현(Action)**
- 기술적 의사결정과 그 이유
- 주요 기술적 챌린지 및 해결 방식

**성과(Result)**
- 정량적 지표 (성능 개선율, 비용 절감, 처리량 등)
- 정성적 성과 (프로세스 개선, 팀 기여 등)

**기술 스택**: Java, Spring Boot, Elasticsearch, Redis...

실무 예시 (TikTok 데이터 시스템 기준)
markdown### TikTok 인플루언서 데이터 수집/검색 시스템 (2024.01 ~ 현재)
> 14,500+ 인플루언서 프로필과 200만 건 이상의 콘텐츠를 실시간 검색 가능한 플랫폼 구축

**상황**
- 마케팅팀의 인플루언서 탐색 시간이 평균 2시간 이상 소요
- 기존 RDB 기반 검색의 응답 속도 3초 이상, 복합 조건 검색 불가

**역할**
- 백엔드 설계 및 개발 전담 (1인)
- 데이터 파이프라인 및 검색 엔진 아키텍처 설계

**핵심 구현**
- Elasticsearch 인덱스 설계: ngram tokenizer 적용으로 부분 문자열 검색 지원
- Alias Switching 전략: 무중단 재인덱싱으로 디스크 사용량 2.5x → 2x 최적화
- 배치 처리 안정화: 데드락 감지 및 트랜잭션 분리로 실패율 0% 달성

**성과**
- 검색 응답 시간: 3초 → 200ms (93% 개선)
- 인플루언서 탐색 시간: 2시간 → 15분 (87% 단축)
- 일 평균 검색 요청 5,000건 안정 처리

**기술 스택**: Java 17, Spring Boot 3.x, Elasticsearch 8.5, Redis, MySQL

포맷 설계 의도
요소목적주의점한 줄 요약빠른 스캐닝, 면접관 관심 유도기술보다 비즈니스 임팩트 중심상황/역할프로젝트 맥락 이해팀 규모, 본인 기여도 명확히핵심 구현기술적 역량 증명Why(설계 의도) 포함 필수성과임팩트 증명가능한 정량화, Before/After
트레이드오프: 상세할수록 깊이 있어 보이지만, 프로젝트당 15~20줄 이내가 가독성 측면에서 적정합니다. 면접에서 확장 설명할 여지를 남겨두는 것도 전략입니다.
```
