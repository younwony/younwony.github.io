Java/Spring 기반 8년차 백엔드 개발자로, 이커머스 플랫폼에서 대용량 데이터 파이프라인과 검색 시스템을 설계·구현해왔습니다.

### 기술적 의사결정 (Why & How)

#### Decision 1: Elasticsearch 도입

```
[Context]
- 초기 수십만 건: RDB 충분
- 200만 건 + 복합 필터: 응답 10초+ 지연

[Options]
A. RDB 인덱스 최적화 → 복합 조건에 한계 (4초로 개선되었으나 부족)
B. Elasticsearch 도입 → 역색인 기반 빠른 검색
C. 신규 DB 도입 → 학습/운영 비용 과다

[Decision]
B 선택 - 구하다 커머스에서 이미 ES 운영 중
동일 기술 스택으로 운영 복잡도 최소화

[Trade-off]
- 데이터 동기화 지연 (Near Real-time)
- ES 운영 노하우 필요 → 기존 경험 활용
```

#### Decision 2: Redis Distributed Lock

```
[Context]
- 멀티 인스턴스 환경에서 동시 수집
- 외부 API Rate Limit 존재

[Solution]
Redis Distributed Lock으로 인스턴스 간 동기화
TTL 60초로 데드락 방지 + 재시도 없이 다음 배치에서 처리 (멱등성)

[Result]
- API Rate Limit 준수
- 중복 데이터 0%
```

### 개선 과정 (Improvement Journey)

| 초기 접근 | 문제 발견 | 개선 적용 |
|----------|----------|----------|
| Batch Size 고정값 설정 | 대용량 처리 시 OOM 발생 | Adaptive Batch Size 도입으로 메모리 안정화 |
| 단순 재시도 로직 | API Rate Limit 증폭 | Exponential Backoff + 실패 유형별 분리 |
| 전체 배치 실패 구조 | 외부 API 장애 시 전체 중단 | 부분 실패 허용 구조로 안정성 확보 |

> **Lesson Learned**: 초기 설계 시 데이터 규모 예측 부족 → 이후 프로젝트에서 Scale-out 전제 설계 원칙 수립

### 정량적 성과

| 지표 | Before | After | 개선율 | 측정 방법 |
|------|:------:|:-----:|:------:|----------|
| 검색 응답 시간 | 10초+ | 1초 이내 | **90%↓** | Elastic APM |
| 데이터 풀 | 10만 명 | 200만 명 | **20배↑** | DB 레코드 수 |
| 일 수집량 | 수십 명 (수작업) | 5,000명+ (자동) | **100배↑** | 배치 로그 분석 |
| 리포팅 업무 | 반나절/주 | 0 (Zero Ops) | **100%↓** | - |

> **ROI 환산**: 수작업 제거로 마케팅팀 월 80+ Man-hours 절감, 대형 캠페인 진행 가능으로 신규 매출 기회 창출

## Technical Decision Archive

> 8년차 개발자의 핵심 역량: **기술 선택의 근거와 Trade-off를 명확히 설명하는 능력**

### Decision Log

| 결정 | 선택지 | 선택 이유 | Trade-off |
|------|--------|----------|-----------|
| **Elasticsearch** | RDB vs ES | 역색인 검색, 기존 운영 경험 활용 | Near Real-time 동기화 지연 |
| **Redis Distributed Lock** | DB Lock vs Redis | 멀티 인스턴스 분산 환경 지원 | 인프라 의존성 증가 |
| **인메모리 캐시** | Redis vs In-Memory | 데이터 작음, 인프라 단순화 | 서버별 캐시 불일치 가능 |
| **ChatOps (Slack)** | Admin Web vs Slack | 구성원 익숙, 채택률 극대화 | Slack 의존성 |
| **@Scheduled + Executor** | Spring Batch vs 직접 구현 | 운영 인프라 단순화, 재시도 제어 용이 | Job 관리 UI 부재 |
| **부분 해시 검증** | 전체 해시 vs 부분 해시 | 빠른 변경 감지, 리소스 절약 | 완전한 검증 대신 속도 우선 |

### Avoided Anti-Patterns (피한 안티패턴)

> "무엇을 아는가"보다 **"무엇을 안 하는가"**를 명확히 하는 사람은 드뭅니다.

| 안티패턴 | 피한 이유 |
|---------|----------|
| 모든 문제를 Kafka로 풀려는 시도 | 이벤트 규모와 복잡도 대비 운영 비용 과다, Exactly Once 처리 요구 없음 |
| 캐시 일관성을 무시한 과도한 Redis 의존 | 인프라 장애 시 전체 시스템 영향, 데이터 크기가 작으면 인메모리로 충분 |
| Admin Web 과도 설계로 운영팀 채택 실패 | 기존 익숙한 도구(Slack) 활용이 채택률 극대화 |
| 배치 실패를 로그로만 남기는 구조 | 장애 인지 지연, Slack 알림으로 즉시 대응 체계 구축 |
| 과도한 추상화로 유지보수 복잡도 증가 | "필요할 때 추상화" 원칙, 최소 3곳에서 사용될 때만 공통화 |
| 실시간 스트리밍 과도 도입 | Near Real-time으로 비즈니스 SLA 충족, 장애 시 복구 복잡성 감소가 더 중요 |
