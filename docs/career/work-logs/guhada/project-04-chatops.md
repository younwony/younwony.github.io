# ChatOps 기반 운영 프로세스 자동화 (Zero Ops)

> 작성일: 2025-12-24
> 소속: (주)구하다 / Tech 팀
> 기간: 2023 - 2024

---

## 개요

Slack 기반 ChatOps 시스템을 구축하여 운영팀 수동 업무를 40%에서 0%로 완전 자동화한 프로젝트.

---

## 배경 및 문제점

### 기존 상황
- 발주·클레임 처리가 이메일, 개인 메신저, 구두 요청으로 분산
- 운영팀이 어드민 접속 → 데이터 확인 → 메신저 공유의 컨텍스트 스위칭 반복

### 문제점
- **소통 채널 파편화**: 정보 누락 및 히스토리 추적 불가
- **운영 비효율**: 반복적인 발주 확인과 수동 처리로 업무 시간의 약 40%가 단순 작업에 소모
- **현업 불만**: "어디로 요청해야 할지 모르겠다", "처리 결과를 즉시 알고 싶다" 지속 접수

### 목표
- 모든 운영 요청 창구를 Slack으로 일원화
- 별도 어드민 접속 없이 메신저 내 업무 완결 'Zero Ops' 환경 구축
- 업무 처리 과정 투명화

## 상세 구현

### 아키텍처/설계

```
┌─────────────────────────────────────────────────────────────┐
│                      Slack Workspace                         │
│  ┌─────────────────────────────────────────────────────┐    │
│  │            Interactive Components                    │    │
│  │         (버튼, 모달, 드롭다운)                       │    │
│  └─────────────────────────┬───────────────────────────┘    │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   Slack Event API                            │
│              (이벤트 수신 + 응답)                            │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                 Spring Boot Application                      │
│  ┌──────────────┬──────────────┬──────────────────────┐     │
│  │   발주 처리   │  클레임 처리  │   실시간 알림        │     │
│  │   승인/반려   │   승인/반려   │   상태 변경 알림     │     │
│  └──────────────┴──────────────┴──────────────────────┘     │
└─────────────────────────────┬───────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                       Database                               │
│                  (상태 변경 + 이력 저장)                     │
└─────────────────────────────────────────────────────────────┘
```

### 기술 스택

| 구분 | 기술 |
|------|------|
| 언어 | Java |
| 프레임워크 | Spring Boot |
| 메시징 | Slack Event API, Interactive Components |
| 데이터베이스 | MySQL |

### 구현 내용

#### 1. 현업 인터뷰 직접 리딩
- 운영팀·유관 부서 인터뷰 주도
- 가장 큰 병목(어드민 접속 → 데이터 확인 → 메신저 공유의 컨텍스트 스위칭) 발굴

#### 2. 사용자 중심 플랫폼 선정
- 구성원이 가장 익숙한 Slack 선택으로 진입 장벽 제거
- 채택률 극대화

#### 3. 협업 워크플로우 설계
- 스레드 기능으로 논의 집중
- 버튼 클릭만으로 업무 종결 가능한 구조

#### 4. ChatOps 시스템 구축
- Slack Event API 기반 Interactive Component(버튼, 모달)
- 발주 승인/반려, 클레임 처리 구현
- 이벤트 발생 시 즉시 알림 + DB 처리 완료

### 기술적 의사결정

| 결정 | 선택 | 이유 |
|------|------|------|
| 플랫폼 | Slack | 구성원이 가장 익숙한 플랫폼, 별도 앱 설치 없이 즉시 도입으로 채택률 극대화 |
| 인터랙션 | Event API + Interactive Component | 버튼 클릭만으로 DB 처리 완료, 컨텍스트 스위칭 제거 |

## 결과 및 성과

### 정량적 성과
- **Zero Ops 달성**: 운영팀 수동 업무 40% → 0% 완전 자동화
- **채널 100% 통합**: 파편화된 채널 일원화로 부서 간 협업 속도 증대
- **업무 투명성 확보**: 모든 요청·처리 내역 Slack 기록으로 히스토리 추적 가능

### 정성적 효과
- 운영팀 고부가가치 업무 집중 가능
- 휴먼 에러 방지로 클레임 감소
- 현업 만족도 향상: "요청 창구 명확", "처리 결과 즉시 확인"
