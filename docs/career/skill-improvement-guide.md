## 핵심 보완 방향 (3줄 요약)

1. **guide.md를 마스터로** 지정하고, 나머지 문서는 참조만 하도록 중복 제거
2. **Trade-off Rule 명문화**: 조직 영향 > 기술 난이도, 재사용성 > 단발성
3. **"쓰는 법"에서 "생략하는 법"으로**: 기술 상세 상한선, 의도적 여백 설계 추가

## 1. guide.md (작성 가이드) 보완점

### 1.1 Trade-off Rule 추가 (최우선)

```markdown
## 최종 판단 기준 (Trade-off Rule)

1. 조직/비즈니스 영향 > 기술 난이도
2. 재사용/확산 가능성 > 단발성 성과
3. 설명이 길어지는 프로젝트는 대표작이 아니다
```

> 이 3줄이 있으면 모든 가이드가 하나의 철학으로 묶임

### 1.2 Action 섹션에 "기각된 대안(Why Not)" 필수화

| 현재 | 보완 |
|------|------|
| "왜 Kafka를 썼나?" | "왜 RabbitMQ나 SQS를 **안 쓰고** Kafka를 썼나?" |

```markdown
## A - Action 작성 규칙

**필수 포함**: 고려했으나 기각한 대안(Rejected Alternatives)과 그 이유 1줄 이상

예시:
- "RabbitMQ 검토 → 메시지 순서 보장 필요로 Kafka 선택"
- "Redis Pub/Sub 검토 → 메시지 영속성 필요로 Kafka 선택"
```

### 1.3 면접 질문 유도 패턴 추가

```markdown
## 면접 질문 유도 패턴

### 의도적 여백 설계
모든 답을 문서에 쓰지 않고, 면접관이 질문하도록 유도

| 패턴 | 예시 |
|------|------|
| 비교 대상 언급만 | "RDB 기반 접근의 한계로 전환 결정" |
| 리스크 존재 암시 | "멀티 인스턴스 환경에서 중복 수집 이슈 발생" |
| 협업 지점만 던지기 | "MD팀과 운영 우선순위 충돌 조율" |
```

### 1.4 짧은 경력 대응 전략 추가

```markdown
## 짧은 경력 대응 전략 (5개월 이하)

**원칙**: "왜 떠났나"보다 "무엇을 남겼나"에 집중

| 피해야 할 표현 | 권장 표현 |
|---------------|----------|
| "짧은 기간이었지만..." | (언급하지 않음) |
| "적응 중이었으나..." | "TDD 문화 정착에 기여" |

**면접 예상 질문 대비**:
- "왜 5개월 만에 떠났나요?"
  → "더 큰 도전 기회 (인플루언서 플랫폼 0→1 구축)"
- "짧은 기간에 무엇을 했나요?"
  → "레거시 시스템에 TDD 도입, 핵심 비즈니스 로직 테스트 커버리지 확보"
```

### 1.5 포지션별 강조 포인트 추가

```markdown
## 포지션별 강조 포인트

| 포지션 | 1순위 강조 | 2순위 강조 | 톤 |
|--------|-----------|-----------|-----|
| 검색/데이터 플랫폼 | ES 경험, 파이프라인 | 대용량 처리 | 기술 깊이 |
| 이커머스 백엔드 | 도메인 경험, 매출 성과 | MD 협업 | 비즈니스 |
| 플랫폼/DevOps | Zero Ops, 자동화 | 모니터링 | 효율성 |
| 테크리드 | 팀 전파, 문화 개선 | 의사결정 | 리더십 |
```

### 1.6 리더십 정량화 가이드 추가

```markdown
## Engineering Culture 기여도 정량화

- 채용 기여: 면접관 참여 횟수
- 온보딩 단축: 신규 입사자 생산 투입 시점 (2주 → 3일)
- 사내 세미나: 주최 횟수, 참여자 수
- 코드 리뷰: 리뷰 건수, 승인율
```

### 1.7 "의도적으로 쓰지 않는 것" 목록 추가

```markdown
## 제외 항목 (면접에서 말할 여지 확보)

- 프레임워크 버전 상세 (Spring Boot 3.2.1 등)
- 라이브러리 내부 구현 설명
- 단순 CRUD/유지보수성 작업
- 설정 파일 변경 수준의 작업
```

```markdown
## JD 분석 로직

단순 키워드 매칭이 아닌 **Problem-Solution Fit** 분석

| 스타트업/수시 | 2페이지 (프로젝트 요약 포함) |

## 3. career.md (경력기술서) 보완점

### 3.1 기술 상세 상한선 규칙 추가

```markdown
## 기술 상세 상한선

- Troubleshooting: 최대 1개 이슈만 상세
- 코드 스니펫: 핵심 설계 판단이 드러나는 1곳만
- Architecture Diagram: 대표 프로젝트 2~3개만 허용
```

### 3.2 Tier 1/Tier 2 프로젝트 분리

```markdown
## 프로젝트 구성 전략

### Tier 1: 서사형 (3개, 각 1-2페이지)
- STAR+I 상세
- 아키텍처 다이어그램
- 트러블슈팅 상세
- 기술 의사결정(WHY) 포함

### Tier 2: 요약형 (2-3개, 각 5-10줄)
- 한 줄 요약 + 핵심 성과 2-3개
- 기술 스택만 나열
- WHY 제외

### 선정 기준

| Tier 1 조건 | Tier 2 조건 |
|-------------|-------------|
| 비즈니스 임팩트 명확 | 기술적 깊이는 있으나 임팩트 약함 |
| 기술 의사결정 복잡 | 단순 구현 위주 |
| 면접 확장 설명 가능 | 추가 질문 가능성 낮음 |
```

### 3.3 코드 스니펫 "의도 주석" 필수화

```markdown
## 코드 스니펫 규칙

**필수**: 이 코드가 해결한 핵심 문제(주석) 포함

```java
// Before: 단순히 락을 거는 코드
// After: 데드락 방지를 위해 TTL 설정 및 멱등성을 고려한 분산락 구현
boolean acquired = lock.tryLock(wait, lease, TimeUnit.SECONDS);
```
```

### 3.4 아키텍처 표현 표준화

```markdown
## 아키텍처 다이어그램 규칙

복잡한 구조는 텍스트 흐름도 우선 사용:

[Input] -> [Process] -> [Output]

또는 Mermaid.js 문법:
```mermaid
graph LR
    A[TikTok API] --> B[Collector]
    B --> C[Elasticsearch]
```
```

### 3.5 실패 회고(Lesson Learned) 추가

```markdown
## Troubleshooting 섹션 구조

### 필수 항목
1. 문제 발견 (어떻게 알게 되었나)
2. 원인 분석 (시도한 가설들)
3. 해결 과정 (시도-실패-성공)
4. **Lesson Learned** (실패를 통해 얻은 인사이트)

### Lesson Learned 예시
- "초기 설계 시 확장성 고려 부족 → 이후 프로젝트에서 Scale-out 전제 설계"
- "모니터링 없이 배포 → APM 필수 적용 원칙 수립"
```

## 4. 세 문서 공통 보완점

### 4.1 SSOT 원칙 명시

```markdown
## 데이터 정합성

모든 수치와 성과는 `docs/career/my_career_data.md`를 원본으로 하며,
절대 임의로 변경하지 않는다.

수치 변경 시 기준:
- 유효숫자 2자리까지 표기
- 반올림 기준 명시
```

### 4.2 이력서-경력기술서 중복 방지

```markdown
## 중복 방지 원칙

두 문서를 함께 제출할 경우, 이력서는 "요약"이 아닌 "다른 관점"을 제공

| 항목 | 이력서 | 경력기술서 | 중복 허용 |
|------|--------|-----------|----------|
| Professional Summary | 3줄 (숫자 강조) | 5줄 (문제→해결 구조) | 표현 차별화 필수 |
| 프로젝트 | 1줄 성과만 | STAR+I 전체 | 금지 |
| 기술 선택 이유 | 제외 | 상세 포함 | 금지 |
| 아키텍처/트러블슈팅 | 제외 | 상세 포함 | 금지 |
| 기술 스택 | 카테고리별 요약 | 프로젝트별 상세 | 허용 |
```

### 4.3 연차 증가 시 업데이트 원칙

```markdown
## 업데이트 원칙

- 수치 추가: 허용
- 프로젝트 수 증가: 금지 (총량 유지)
- 같은 유형 프로젝트: 교체만 허용
```
