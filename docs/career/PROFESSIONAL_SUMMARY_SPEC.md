# Professional Summary 작성 명세서

> **목적**: 8년차 시니어 백엔드 개발자의 Professional Summary 작성 가이드
> **핵심 원칙**: 한 문장에 **[해결 방법] + [정량 성과]**를 자연스럽게 녹여서 작성

---

## 1. 작성 원칙

### 1.1 8년차 개발자의 포지셔닝

| 구분 | 설명 |
|------|------|
| **포지션** | Senior IC(Individual Contributor) → Tech Lead 전환기 |
| **증명 포인트** | "무엇을 할 줄 안다" → **"비즈니스 문제를 기술로 어떻게 해결했는지"** |
| **중요성** | 채용 담당자가 가장 먼저 읽는 **3초의 승부처** |

### 1.2 성공하는 이력서의 특징

```
❌ 나열식: "Kafka를 사용해봄"
✅ 임팩트 중심: "MSA 환경의 데이터 정합성 문제를 해결하기 위해 Kafka를 도입하여 처리량 300% 개선"
```

**핵심 요소:**
- 기술 도입의 **이유(Why)** 명시
- **결과(Result)**를 수치로 표현
- 기술적 의사결정의 배경과 성과를 유기적으로 연결

---

## 2. 8년차 핵심 역량 (Core Competencies)

기업이 8년차에게 기대하는 3가지 역량:

| 역량 | 세부 내용 |
|------|----------|
| **아키텍처 설계** | 대용량 트래픽 처리, 레거시 리팩토링, MSA 전환 경험 |
| **비즈니스 기여** | 매출 증대, 비용 절감, 운영 효율화 등 지표 개선 |
| **소프트 스킬** | 주니어 멘토링, 코드 리뷰 문화 정착, 유관 부서 커뮤니케이션 |

> 세 가지 요소가 균형 있게 배치되어야 **"믿고 맡길 수 있는 시니어"** 인상

---

## 3. 작성 구조

### 3.1 기본 공식

```
[연차/직무] + [핵심 기술 스택] + [가장 큰 성과/강점] + [직무 태도]
```

### 3.2 구성 요소

| 순서 | 요소 | 설명 |
|------|------|------|
| 1 | **도입부** | 연차, 직무, 핵심 도메인 경험 (1문장) |
| 2 | **차별점** | 단순 구현이 아닌 기술적 의사결정 관점 (1문장) |
| 3 | **핵심 성과** | 정량 수치가 포함된 대표 성과 3-4개 |

### 3.3 분량 가이드

| 버전 | 분량 | 용도 |
|------|------|------|
| 간결 버전 | 3문장 | 이력서 상단, 링크드인 요약 |
| 표준 버전 | 1-2문단 | 이력서 Professional Summary 섹션 |
| 상세 버전 | 3문단 | 자기소개서, 포트폴리오 |

## 4. 유형별 템플릿

### Type A: 대용량 트래픽/성능 최적화 강조형 (기술 중심)

**적합 대상**: 대규모 시스템 운영, 성능 최적화, 인프라 경험이 풍부한 경우

```
"Java/Spring Boot 기반의 8년차 백엔드 개발자입니다.
일일 트래픽 000만 건의 대규모 이커머스 시스템을 운영하며,
레거시 모놀리식 아키텍처를 MSA로 성공적으로 전환하여 시스템 안정성을 99.9%까지 끌어올린 경험이 있습니다.
[기술 스택]을 활용한 [성과]에 강점이 있으며,
[최적화 영역]을 통해 [정량 성과]를 보유하고 있습니다."
```

### Type B: 비즈니스 임팩트/문제 해결 강조형 (성과 중심)

**적합 대상**: 비즈니스 지표 개선, 자동화, 운영 효율화 경험이 풍부한 경우

```
"비즈니스 문제를 기술로 해결하는 데 주력하는 8년차 백엔드 엔지니어입니다.
[도메인]에서 [시스템/기능]을 자동화하여 운영 리소스를 00% 단축하고,
휴먼 에러를 'Zero'화한 경험이 있습니다.
[기술 스택]을 주력으로 사용하며, 복잡한 비즈니스 로직을 간결한 코드로 구현하는 것을 즐깁니다."
```

### Type C: 리더십/팀 빌딩 강조형 (매니징/리드급 희망 시)

**적합 대상**: 팀 리딩, 개발 문화 개선, 멘토링 경험이 풍부한 경우

```
"주도적인 개발 문화를 지향하는 8년차 백엔드 리드 개발자입니다.
0명 규모의 개발팀을 리딩하며 코드 리뷰 문화 정착과 TDD 도입을 주도하여
버그 발생률을 00% 감소시켰습니다.
[기술 스택]에 전문성이 있으며, 주니어 개발자 멘토링을 통해
팀 전체의 기술 역량을 상향 평준화하는 데 보람을 느낍니다."
```

## 5. 성과 기술 방법론 (STAR 기법)

### 5.1 STAR 구조

| 단계 | 항목 | 8년차 포인트 |
|------|------|-------------|
| **S** | Situation (배경) | 문제 상황, 제약 조건 명시 |
| **T** | Task (과제) | 해결해야 할 목표 (정량 목표 포함) |
| **A** | Action (해결) | **기술적 난이도 + 주도적 역할** 강조 |
| **R** | Result (성과) | **정량 수치**로 표현 |

### 5.2 Action 작성 시 필수 요소

```
[기술적 의사결정 배경] + [구체적 해결 방법] + [본인의 역할]
```

**예시:**
- ❌ "Elasticsearch를 도입함"
- ✅ "200만 건 복합 조건 검색에서 RDB 타임아웃 발생 → 역색인 기반 Elasticsearch 도입으로 10초+ → 1초 이내 달성"

## 6. 정량 성과 표현 가이드

### 6.1 성과 유형별 표현

| 유형 | 표현 패턴 | 예시 |
|------|----------|------|
| 성능 개선 | N배 개선, N% 단축 | "검색 성능 10배 개선 (10초+ → 1초 이내)" |
| 비용 절감 | N% 절감, N원 절약 | "AWS 비용 80% 절감" |
| 운영 효율 | N% → 0% 자동화 | "운영 업무 40% → 0% 완전 자동화" |
| 매출 기여 | N% 상승, N억 달성 | "가격 경쟁력 확보로 매출 10% 상승" |
| 규모 확장 | N건, N배 증가 | "데이터 풀 10만 → 200만 (20배 확대)" |

### 6.2 Before/After 표현

```
✅ 권장: "10초+ → 1초 이내", "40% → 0%", "수동 → 자동화"
❌ 지양: "많이 개선됨", "효율이 높아짐", "빨라짐"
```

## 7. 체크리스트

### 작성 완료 후 검증

- [ ] 한 문장에 [해결 방법 + 정량 성과]가 포함되어 있는가?
- [ ] 기술 도입의 **Why**가 명확한가?
- [ ] 정량 수치(%, 배수, 시간)가 포함되어 있는가?
- [ ] 8년차 시니어로서의 **기술적 의사결정**이 드러나는가?
- [ ] 비즈니스 임팩트가 명확히 표현되어 있는가?
- [ ] 3초 안에 핵심 역량이 파악되는가?
