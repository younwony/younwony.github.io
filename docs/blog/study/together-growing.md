## 1부 - 자라기

**작성일**: 2022.08.28

### 1. 학습 프레임 vs 실행 프레임

#### 학습 프레임 (Learning Frame)
- **목표**: 실력 향상, 성장
- **관점**: 현재 상태는 일시적
- **태도**: 실수를 배움의 기회로 활용
- **결과**: 장기적 성장

```
학습 프레임을 가진 사람:
"이번 프로젝트에서 새로운 기술을 배워야겠다"
"실수했지만 다음엔 더 잘할 수 있을 것이다"
```

#### 실행 프레임 (Performance Frame)
- **목표**: 당장의 성과
- **관점**: 현재 상태가 고정적
- **태도**: 실수를 숨기려 함
- **결과**: 단기적 성과, 장기적으론 정체

```
실행 프레임을 가진 사람:
"이 기술은 이미 아니까 다른 것을 해야지"
"실수하면 안 돼, 완벽해야 해"
```

### 2. 달인이 되는 비결

#### 의도적 수련 (Deliberate Practice)
1. **구체적 목표 설정**
   - 막연한 "잘하기"가 아닌 구체적 목표
   - 예: "이번 달에 알고리즘 문제 30개 풀기"

2. **즉각적 피드백**
   - 내가 한 행동의 결과를 빠르게 확인
   - 예: 테스트 주도 개발(TDD)

3. **불편함을 감수**
   - 현재 능력의 경계선에서 연습
   - 너무 쉬운 것도, 너무 어려운 것도 아닌 적절한 난이도

#### 1만 시간의 법칙?
- **잘못된 해석**: 단순히 시간만 채우면 달인
- **올바른 해석**: 의도적 수련 1만 시간이 필요

> "그냥 10년 일했다고 전문가가 되는 게 아니다.
> 의도적 수련을 10년 해야 전문가가 된다."

### 3. 제자리걸음의 원인

#### 자동화의 함정
- 업무가 자동화되면 더 이상 배우지 않음
- 편안한 상태 = 성장 정체

#### 탈출 방법
1. **의도적으로 불편함 선택**
   - 새로운 기술 시도
   - 다른 접근법 시도

2. **피드백 루프 짧게**
   - 짧은 주기로 결과 확인
   - 빠른 수정과 개선

### 4. 프로그래밍 언어 학습 방법론

#### 잘못된 방법
- 문법만 달달 외우기
- 튜토리얼만 따라하기
- 한 언어만 깊게 파기

#### 올바른 방법
1. **여러 언어 학습**
   - 다양한 패러다임 경험 (객체지향, 함수형, 절차적)
   - 사고의 폭 확장

2. **작은 프로젝트 만들기**
   - 실제 문제 해결
   - 실전 경험 축적

3. **언어 간 비교**
   - 같은 문제를 다른 언어로 해결
   - 장단점 파악

### 5. 실수 관리

#### 실수에 대한 두 가지 태도

**부정적 태도** (실행 프레임)
- 실수를 숨김
- 방어적 태도
- 같은 실수 반복

**긍정적 태도** (학습 프레임)
- 실수를 공유
- 배움의 기회로 활용
- 개선과 성장

#### 심리적 안전감
- 실수해도 처벌받지 않는 환경
- 실수를 공유하고 배우는 문화
- 구글의 연구: 성공하는 팀의 핵심 요소

### 6. 선생과 전문가

#### 가르치는 것이 배우는 것
- 다른 사람을 가르치면 자신도 더 깊이 이해
- "설명할 수 없으면 제대로 이해한 것이 아니다" - 아인슈타인

#### 짝 프로그래밍의 효과
- 서로 가르치고 배우는 과정
- 코드 품질 향상
- 지식 공유

## 2부 - 함께

**작성일**: 2022.08.28

### 1. 협력의 중요성

#### 혼자보다 함께
- 1 + 1 > 2의 시너지 효과
- 서로 다른 관점의 조합
- 지식과 경험의 공유

### 2. 조엘 테스트 (Joel Test)

개발팀의 수준을 평가하는 12가지 질문:

1. ✅ 소스 관리 시스템을 사용하는가?
2. ✅ 하루에 한 번씩 빌드할 수 있는가?
3. ✅ 데일리 빌드를 하는가?
4. ✅ 버그 추적 시스템을 사용하는가?
5. ✅ 새 코드를 작성하기 전에 버그를 수정하는가?
6. ✅ 최신 일정표를 유지하는가?
7. ✅ 스펙(명세)이 있는가?
8. ✅ 프로그래머가 조용한 환경을 가지는가?
9. ✅ 돈이 허락하는 한 최고의 도구를 사용하는가?
10. ✅ 테스터가 있는가?
11. ✅ 프로그래머를 채용할 때 코딩 테스트를 하는가?
12. ✅ 복도 사용성 테스트를 하는가?

**점수 해석**:
- 12점: 완벽
- 11점: 참 좋음
- 10점 이하: 심각한 문제 있음

### 3. 협력을 통한 추상화

#### 페어 프로그래밍의 장점
1. **코드 품질 향상**
   - 즉각적인 코드 리뷰
   - 버그 조기 발견

2. **지식 공유**
   - 암묵지의 명시지화
   - 팀 전체 역량 향상

3. **추상화 능력 향상**
   - 다른 사람에게 설명하는 과정에서 추상화 발생
   - 더 나은 설계

### 4. 신뢰 구축

#### 신뢰의 중요성
- 협업의 기반
- 심리적 안전감 제공
- 효율적 의사소통

#### 신뢰 구축 방법
1. **투명성**: 정보 공개
2. **일관성**: 말과 행동의 일치
3. **책임**: 약속 이행
4. **공감**: 상대방 이해

### 5. 구글의 탁월한 팀 비결

#### 프로젝트 아리스토텔레스
구글이 수백 개 팀을 연구한 결과, 성공하는 팀의 핵심 요소:

1. **심리적 안전감** (Psychological Safety) ⭐ 가장 중요
   - 실수해도 괜찮은 분위기
   - 위험을 감수할 수 있는 환경

2. **신뢰성** (Dependability)
   - 동료를 믿을 수 있음
   - 약속 이행

3. **구조와 명확성** (Structure & Clarity)
   - 명확한 역할과 목표
   - 체계적인 프로세스

4. **의미** (Meaning)
   - 일의 의미 부여
   - 개인적 중요성

5. **영향** (Impact)
   - 일이 변화를 만든다는 믿음
   - 기여의 가시성

### 6. 프로젝트 확률론

#### 불확실성 다루기
- 개발 프로젝트는 본질적으로 불확실
- 확률적 사고 필요

#### 린 스타트업 방식
1. **가설 수립**
2. **빠른 실험**
3. **측정 및 학습**
4. **피벗 또는 유지**

## 3부 - 애자일

**작성일**: 2022.08.28

### 1. 애자일의 철학

#### 애자일 선언문 (Agile Manifesto)

**가치**:
- 프로세스와 도구보다 **개인과 상호작용**
- 포괄적인 문서보다 **작동하는 소프트웨어**
- 계약 협상보다 **고객과의 협력**
- 계획을 따르기보다 **변화에 대응**

#### 애자일의 핵심
- 작동하는 소프트웨어를 자주 출시
- 변화를 환영하는 태도
- 비즈니스와 개발자의 긴밀한 협력

### 2. "고객에게 매일 가치를 전하라"

#### 작은 배포의 힘
- 큰 배포: 위험 높음, 피드백 늦음
- 작은 배포: 위험 낮음, 피드백 빠름

#### 지속적 전달 (Continuous Delivery)
1. **자동화된 테스트**
2. **자동화된 배포**
3. **작은 단위의 변경**
4. **빠른 피드백**

### 3. 불확실성 대응 전략

#### 예측보다 적응
- 미래를 정확히 예측 불가능
- 변화에 빠르게 적응하는 능력이 중요

#### 짧은 피드백 루프
```
계획 → 실행 → 측정 → 학습 → (반복)
```

- 주기가 짧을수록 학습 속도 빠름
- 실패 비용 감소

### 4. 애자일 도입 성공 요인

#### 실패하는 애자일 도입
- 형식만 따라하기 (Cargo Cult Agile)
- 도구와 프로세스에만 집중
- 조직 문화 무시

#### 성공하는 애자일 도입
1. **문화 변화**
   - 심리적 안전감 조성
   - 실험과 학습 장려

2. **점진적 도입**
   - 작은 것부터 시작
   - 성공 경험 축적

3. **지속적 개선**
   - 회고(Retrospective)
   - 학습 조직화

## 핵심 교훈 정리

### 1. 자라기 (학습)
> "학습 프레임을 가져라. 실수를 배움의 기회로 삼아라."

- 의도적 수련
- 피드백 루프 짧게
- 불편함을 감수

### 2. 함께 (협력)
> "혼자 가면 빨리 가지만, 함께 가면 멀리 간다."

- 심리적 안전감
- 페어 프로그래밍
- 지식 공유

### 3. 애자일 (변화 대응)
> "계획을 따르기보다 변화에 대응하라."

- 작은 배포
- 빠른 피드백
- 지속적 개선

## 실무 적용 가이드

### 개인 차원
1. ✅ 학습 프레임으로 전환
   - "이번 프로젝트에서 뭘 배울까?" 질문하기
   - 실수 기록 및 회고

2. ✅ 의도적 수련
   - 약간 어려운 문제에 도전
   - 즉각적 피드백 받기

3. ✅ 다른 사람 가르치기
   - 블로그 작성
   - 스터디 참여

### 팀 차원
1. 📋 심리적 안전감 조성
   - 실수를 비난하지 않는 문화
   - 질문하기 쉬운 분위기

2. 📋 페어/몹 프로그래밍 도입
   - 주 1회 페어 프로그래밍
   - 지식 공유 세션

3. 📋 짧은 피드백 루프
   - 스프린트 단축 (2주 → 1주)
   - 데일리 스탠드업

### 조직 차원
1. 🎯 조엘 테스트 실시
   - 현재 수준 파악
   - 개선 계획 수립

2. 🎯 애자일 점진적 도입
   - 스크럼/칸반 시도
   - 회고 문화 정착

3. 🎯 학습 조직화
   - 사내 스터디 지원
   - 컨퍼런스 참가 장려

## 인상 깊은 구절

> "당신이 제자리걸음인 이유는 당신이 뛰어나서가 아니라, 학습 프레임이 아니어서 입니다."

> "프로그래밍은 혼자 하는 작업이 아닙니다. 소프트웨어 개발은 협력적 게임입니다."

> "애자일은 도구가 아니라 철학입니다. 형식을 따라하는 것이 아니라 원칙을 이해하고 적용하는 것입니다."

> "실수를 저지르는 것보다 더 나쁜 것은, 실수로부터 배우지 않는 것입니다."
