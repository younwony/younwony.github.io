## 0. 내용 참고
`docs/career/my_career_data.md` 파일의 내용을 참고하여, 경력직 백엔드 개발자 이력서 작성법에 대한 가이드 문서를 작성합니다.

## 📌 핵심 원칙

### 마인드셋의 전환

| 변경 전 | 변경 후 |
|---------|---------|
| 개발자(Coder) | **전략가(Strategist)** - 기술로 비즈니스 문제를 해결하는 사람 |
| "담당했습니다" | **"달성/개선했습니다"** - 능동적인 성과 강조 |
| 모든 경험 나열 | **상위 20% 핵심 프로젝트 집중** (파레토 법칙) |

### 핵심 메시지

> **"무엇을 했다(What)"가 아닌 "왜(Why) → 어떻게(How) → 성과(Impact)"를 보여주세요.**

7년차 개발자에게 채용담당자가 보고 싶은 건 "코딩 능력"이 아니라
**"문제 정의 → 기술적 판단 → 비즈니스 기여"의 사고 구조**입니다.

## 🎯 작성 프레임워크: STAR+I 모델

모든 경력 기술은 아래 구조를 따릅니다:

| 요소 | 설명 | 핵심 질문 |
|------|------|----------|
| **S** (Situation) | 배경·문제·제약 | 어떤 어려움이나 비효율이 있었는가? |
| **T** (Task) | 과제/목표 | 해결해야 할 과제는 무엇인가? |
| **A** (Action) | 기술적 의사결정 (Why & How) | **왜 그 기술을 선택했으며, 어떻게 설계했는가?** |
| **R** (Result) | 정량적 성과 | 수치로 증명된 결과는 무엇인가? |
| **I** (Impact) | 비즈니스/조직적 효과 | 어떤 비즈니스 임팩트를 만들었는가? |

### 문장 템플릿

```
"[제약/문제](S) 속에서 [기술적 의사결정](A)으로 [정량적 성과](R) 달성,
[비즈니스/운영 효과](I)"
```

### 예시

> "API Limit 25k 제약(S) 속에서 우선순위 큐 기반 가격조정 로직 설계(A)로
> 핵심 상품 대응률 90% 달성(R), 운영팀 수작업 완전 제거로 리소스 집중도 향상(I)"

## ❌ Bad vs ✅ Good 표현 사례

| 구분 | ❌ 나쁜 표현 (단순 나열) | ✅ 좋은 표현 (문제 해결 & 성과 중심) |
|------|------------------------|-----------------------------------|
| **데이터 수집** | TikTok 데이터 크롤링 시스템 개발 담당 | TikTok 크롤링 시스템 리팩토링 및 단일 책임 원칙 적용으로 유지보수성 향상, **일일 데이터 처리량 10만→25만 건(2.5배) 확장** |
| **DB 최적화** | MySQL 쿼리 최적화 수행 | LIKE 검색을 **Full-Text Index(ngram parser)**로 전환하여 한글 검색 응답 시간을 **3초→0.2초로 93% 단축**, 사용자 이탈 방지 |
| **도구 개발** | Spring Boot 기반 백엔드 개발 | Java Spring Boot 기반 MCP 서버 설계로 Claude Desktop 연동 자동화 구현, **반복 수작업 시간 주당 5시간 절감** |
| **기술 도입** | Redis Lock 적용 | 동시성 이슈(Race Condition) 해결을 위해 Redis Distributed Lock 도입하여 **데이터 정합성 100% 확보 및 재고 오류 제로화** |
| **검색 기능** | ElasticSearch 구축 | 검색시간 **1.2초→200ms(ngram/cache)** 개선, 이탈률 15% 감소 |
| **배치 처리** | 배치 처리 구현 | Failover 포함 운영 중단 0건 달성, SLA 99.9% 유지 |

## 🚀 7년차를 위한 'Plus Alpha' 전략

시니어/리드급으로 평가받기 위해 반드시 포함해야 할 3가지 요소:

### 1. 기술적 트레이드오프(Trade-off)와 의사결정 근거

> 시니어 개발자는 정답만 맞히는 사람이 아니라, **'선택의 이유'**를 아는 사람입니다.

**작성 방법:**
- 단순히 "A 기술을 썼다"가 아니라
- **"B와 A를 비교했으나, 당시 트래픽 패턴과 리소스 제약을 고려하여 A를 선택했다"**

**의사결정 근거 키워드:**
- API Limit / Rate Limit
- SLA / RPO / RTO
- Failover / 장애 대응
- Infra Cost / 운영 비용
- Operational Risk / 운영 리스크

**예시:**
> "Batch 대신 Scheduler를 선택한 이유는 운영 리소스 제약과 API Limit을 고려한 의사결정"

### 2. 팀과 조직에 미친 영향력 (Engineering Culture)

> 7년차는 혼자 잘하는 것을 넘어, **팀 전체의 생산성을 높여야** 합니다.

**포함해야 할 경험:**
- 코드 리뷰 문화 정착
- 신규 입사자 온보딩 문서화
- 사내 기술 세미나 주최
- 공통 모듈 개발을 통한 팀 리소스 절감
- 주니어 멘토링

**예시:**
> "팀 내 반복되는 배포 실수를 방지하기 위해 CI/CD 파이프라인에 자동 테스트 단계를 의무화하고,
> 배포 시간을 **30분→5분으로 단축**하여 개발 생산성 향상"

### 3. 비즈니스 언어로 소통하는 능력

> 개발자가 아닌 PM, 기획자, 운영팀과의 **협업 능력**을 보여주세요.

**작성 포인트:**
- 기술 용어를 모르는 동료에게 어떻게 문제를 설명하고 설득했는지
- 운영팀의 고충(Pain Point)을 기술로 어떻게 덜어주었는지

**예시:**
> "마케팅/MD팀과 협의하여 '가격 추종 범위(4%)', '예외 상품군' 정책을 확정하고,
> MD팀이 예외 케이스만 집중 관리할 수 있도록 업무 플로우 설계"

## 📝 작성 예시

### Case 1: 대용량 데이터 처리 (인플루언서 플랫폼)

#### 📌 Situation (배경)
- 기존 단일 스레드 기반 수집기로는 시간당 처리량이 100건에 불과해 캠페인 확장 불가
- 외부 API(TikTok 등)의 엄격한 Rate Limit으로 무작정 요청 증가 불가
- 200만 건 이상 데이터에서 복합 조건 검색 시 RDB 쿼리 타임아웃 발생 (평균 10초 이상)

#### 🎯 Task (과제)
- API 차단 없이 수집 속도를 기존 대비 100배(10,000건/h) 이상으로 향상
- CDN 링크 만료로 인한 이미지 유실 0% 달성
- 200만 건 규모 실시간 검색 제공

#### ⚡ Action (해결) - Why & How
- **병렬 아키텍처 설계**: ThreadPoolTaskExecutor 커스텀으로 I/O Blocking 최소화
- **안정성 확보**: Resilience4j RateLimiter + Backoff Retry로 API 임계치 내 최대 효율 튜닝
- **분산 제어**: Redis Distributed Lock으로 멀티 인스턴스 환경 중복 수집 방지
- **검색 엔진 최적화**: Elasticsearch 도입으로 복합 조건 검색 100배 개선

#### ✅ Result (성과)
- 수집 속도 **99% 단축** (100명/h → 10,000명/h)
- 데이터 풀 **20배 확대** (10만 → 200만 명)
- 검색 속도 **10초 → 0.1초** (100배 개선)
- 리포팅 **Zero Ops** 달성

#### 💡 Impact (비즈니스 효과)
- 마케팅 커버리지 극대화로 대형 캠페인 진행 가능
- 마케팅팀 리포팅 수작업 완전 제거

### Case 2: 동적 가격 시스템 (Dynamic Pricing)

#### 📌 Situation (배경)
- 네이버 쇼핑 최저가 진입 필수, but 수동 대응으로는 경쟁 속도 미달
- 네이버 API 일일 호출 한도 25,000건 제약
- 전체 상품 실시간 대응 불가능한 딜레마

#### 🎯 Task (과제)
- API 제약 내에서 핵심 상품 최저가 방어 알고리즘 구현
- 마진율 4% 보장 범위 내 안전장치 마련
- 자동+수동 하이브리드 운영 구조 설계

#### ⚡ Action (해결) - Why & How
- **비즈니스 규칙 설계 리딩**: 마케팅/MD팀과 협의하여 가격 추종 범위(4%), 최소 마진 기준 정의
- **API Quota 최적화**: 매출 기여도/노출 중요도 기반 우선순위 스케줄링
- **하이브리드 운영 구조**: 시스템 자동 처리 + MD팀 수동 처리 통합, 예외 케이스 Admin 뷰 제공
- **성과 검증 환경**: GA/BigQuery 연동으로 정량 분석 환경 구축

> **Tech Decision**: "Batch 대신 Scheduler 선택 - 운영 리소스 제약과 API Limit 고려"

#### ✅ Result (성과)
- **매출/트래픽 각각 10% 상승** (데이터 검증 완료)
- 운영 효율 혁신: 단순 반복 업무 시스템 전담
- 의사결정 체계 확립: 정기 협의 체계로 가격 정책 신속 대응

#### 💡 Impact (비즈니스 효과)
- 가격 경쟁력 확보로 매출 방어
- MD팀은 전략 상품·예외 케이스에만 집중하는 고효율 구조 정착

### Case 3: 운영 자동화 (ChatOps) - Zero Ops 달성

#### 📌 Situation (배경)
- 발주/클레임 처리가 이메일, 메신저, 구두로 분산 → 정보 누락, 히스토리 추적 불가
- 운영팀 업무 시간의 **40%가 단순 반복 작업**에 소모
- "어디로 요청해야 할지 모르겠다" 현업 불만 지속

#### 🎯 Task (과제)
- 모든 운영 요청 창구를 Slack으로 일원화
- 운영팀 개입 없는 **'Zero Ops'** 환경 구축
- 업무 처리 과정 투명화

#### ⚡ Action (해결) - Why & How
- **현업 인터뷰 직접 리딩**: 운영팀·유관 부서 인터뷰 주도, 병목(컨텍스트 스위칭) 발굴
- **사용자 중심 플랫폼 선정**: 구성원이 가장 익숙한 Slack 선택, 채택률 극대화
- **ChatOps 시스템 구축**: Slack Event API 기반 버튼/모달로 발주 승인/반려 구현

#### ✅ Result (성과)
- 운영팀 수동 업무 **40% → 0%** (Zero Ops)
- 소통 채널 **100% 통합**
- 모든 요청·처리 내역 Slack 기록으로 히스토리 추적 가능

#### 💡 Impact (비즈니스 효과)
- 운영팀이 고부가가치 업무에 집중 가능
- 부서 간 협업 속도 증대 및 휴먼 에러 방지

### Step 3: 체크리스트

- [ ] my_career_data.md의 변경 사항만 반영되었는가?
- [ ] 변경되지 않은 프로젝트는 그대로 보존되었는가?
- [ ] 10개 파일 모두 업데이트되었는가?

**❌ 하지 말 것:**
- 전체 내용 재작성
- 변경되지 않은 프로젝트 수정

- 사용하지 않은 기술 추가

**✅ 해야 할 것:**
- `git diff`로 변경사항 먼저 확인
- 변경된 프로젝트/섹션만 정확히 업데이트
